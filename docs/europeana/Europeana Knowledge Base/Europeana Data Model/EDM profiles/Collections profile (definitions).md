
# Collections profile (definitions)

The EDM Collection Profile provides a structured approach to representing curated cultural heritage collections in Europeana, ensuring that they remain visible, interpretable, and meaningful within large-scale aggregated environments.

# **Introduction and scope**

In the aggregation model used to submit data to Europeana, the totality of a curated collection submitted by a data provider may be difficult to find in a large, aggregated dataset.

‚ÄúCollection structures provide the organizational and intellectual context important to researchers, and collection descriptions provide information needed by users for interpreting the relevance and significance of individual items for their purposes.‚ÄĚ

This profile, based on a [whitepaper on Collections](http://hdl.handle.net/2142/45860), is defined to enable the description of collections in the context of the Europeana Data Model by reusing and extending the classes and properties available. It will allow data providers to supply descriptions of the identifiable collections they have submitted to Europeana. It will also support the definitions of collections (or user-set) of Europeana objects created by end-users. The profile draws on the work cited above and defines the classes and properties that can be used to represent collections in the Europeana system. Besides the¬† namespaces already used by EDM, this profile uses the DC Collection Description Application Profile (<http://purl.org/cld/terms/>).

This document does not take any position on where the values for the properties originate from (data providers, Europeana, other partners).

# **Terminology**

**Collection:** A collection is a group of objects gathered together for some intellectual, artistic, or curatorial purpose[^1]. In the context of this profile the Collection class represents the original collection of cultural heritage objects (probably physical objects but also born digital where appropriate).¬† This definition is visualised in the diagram below in the relationship between the ProvidedCHO and the Collection.

**Dataset:**¬† A dataset for Europeana is an Information Package or a collection of data. A Europeana dataset can be about a certain topic, originate from a certain source or process and is aggregated by a certain custodian.¬† A Europeana dataset can be the representation of one existing collection held by an institution or several collections from different sources aggregated together.¬† The correspondence between datasets and collections cannot be assumed to be one-to-one.

**Provider:**¬† An organisation that collects, formats and manages data from Data Providers before submitting data to <http://Europeana.eu>.

**Data Provider**:¬† An organisation that submits data to <http://Europeana.eu>.¬† Data providers may submit data directly to Europeana but are more likely to use the services of a Provider.

**User set:** A set is a collection of Europeana objects gathered together by a user for some intellectual, artistic, or curatorial purpose (including, for example, a list of favourite items).

# **Representation of user sets**

For the specific case of representing collections created by users ("user sets"), Europeana uses the following fields from the Collection class: `dcterms:title`, `dcterms:description`, `dcterms:creator`, `dcterms:contributor`, `dcterms:created`, `dcterms:modified`,¬† `rdfs:isDefinedBy`, `rdf:type`, `edm:visibility`.

A subset of these properties are used in a specific way, which we indicate below:

|                       |                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
|:----------------------|:-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Property**          | **Usage note**                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `dcterms:creator`     | This property must refer to Agents as defined in Section 2.2. of the EDM Annotations Profile.¬† <br/> This specification expects exactly one `dcterms:creator` to be present for a user set: this will be the entity technically authorized to make edits to the set. We expect cases of "multiple creators" to be represented using `dcterms:contributor` statements. This would be the case for example when the agent responsible for creating the set is replaced by another one, or when several project partners help a registered API user to assemble the relevant items (this registered API user acting as an editor on behalf of all others) |
| `dcterms:contributor` | Refers to the registered users which have requested changes to the set. The API will list the URIs of all users that co-created the set.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |

| `edm:pinned`          | (only usable and mandatory for "entity best item sets") This property represents the number of pinned items in the user set. User sets may be defined in four ways. They can be ‚Äėclosed sets‚Äô that are made out of a predefined list of items, or they can be ‚Äėopen sets‚Äô that are defined by a query, Open sets correspond to the `edm:DynamicCollection` type, and they must always contain the property `rdfs:isDefinedBy` that defines a search request to the Search API (as described in Section 2). Closed sets may be of three types: `edm:Collection, http://rdfs.org/sioc/types#BookmarksFolder` and `edm:EntityBestItemsSet`. They must not contain the `rdfs:isDefinedBy`¬† property, and must list the items within a collection (as described in Section 6).

The User Sets API specification does not use the `edm:Aggregation` construct that is used in Europeana for handling digital representations and provenance. The justification is that for the (Europeana) User Sets these requirements are less relevant. Nevertheless, Aggregations will become more interesting once Europeana starts integrating User Sets from different sources (for example, partner APIs).

This representation of user sets requires introducing three additional subclasses of `edm:Collection` (two of them in the EDM namespace). They are the following:

# **Representing the membership of objects in collections**

The original Collections whitepaper (see footnote 1) as well as earlier versions of this profile suggested to introduce a new EDM property, `edm:isGatheredInto`, in order to relate instances of `edm:ProvidedCHO` to instances of `edm:Collection` they belong to. (proposed definition was: "The identifier of the Collection, the CHO is a member of").

In addition, the property `edm:isNextInSequence` has been suggested in earlier versions of this profile (but *not* in the Collections whitepaper) to represent the order of items within a collection. However using the property on a specific `edm:ProvidedCHO` misses the collection context. A `edm:ProvidedCHO` may belong to several Collections, each with its order. The requirement would therefore not be entirely correctly met.

For the moment we have decided to rely on the Activity Streams mechanism to represent lists of items and possible pagination of these, as specified in the Europeana User Set API. The listing of members is (currently always) represented using pages as defined by Activity Streams (ie. `as:OrderedCollectionPage`). A page will be composed of an ordered list of items using the `as:items` property.

> [!IMPORTANT]
> The choice of using this pagination mechanism was largely motivated by the fact that a collection can be composed of a large number of items, However we are not sure yet that this modeling is appropriate (as it offers two options to represent the same membership information) and we may revise it later.
